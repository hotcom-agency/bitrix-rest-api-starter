# Bitrix REST API

Стартер-проект для разработки REST API на базе Bitrix CMS.

## Структура проекта

```
.
├── docker/                   # Конфигурации Docker-контейнеров
├── scripts/                  # Скрипты автоматизации
├── config/                   # Шаблоны конфигурационных файлов
├── local/                    # Приложение
│   ├── app/                  # Controllers, Services, Dto, Helpers
│   ├── migrations/           # Миграции проекта
│   └── routes/               # Маршруты API
├── bitrix/                   # Ядро Bitrix
├── upload/                   # Загрузки и медиафайлы
├── composer.json             # Зависимости
├── docker-compose.yml        # Конфигурация Docker-приложения
└── Makefile                  # Команды управления
```

## Установка (Docker)

**Требования:**
- Docker Engine ≥ 20.10
- Docker Compose ≥ 2.0
- GNU Make

### Подготовка

Настройте файлы конфигурации под ваше окружение:

- `.env` - переменные окружения
- `Makefile` - команды управления
- `docker-compose.yml` - конфигурация контейнеров

<br>

```bash
cp Makefile.example Makefile && \
cp .env.example .env && \
cp docker-compose.example.yml docker-compose.yml && \
cp docker-compose.dev.example.yml docker-compose.dev.yml
```

### Варианты установки

1. **Чистая установка через `bitrixsetup.php`:**
   - На этапе «Обновление продукта» могут появляться уведомления: «Внимание! На данном шаге произошла ошибка установки продукта» — нажмите **Повторить шаг**.
   - На этапе «Выберите решение для установки» **завершите установку**.

2. **Восстановление из резервной копии через `restore.php`.**

<br>

```bash
make setup-docker
```

### Постустановка (обязательно)

Завершение установки: коррекция прав, проверка конфигов, удаление неиспользуемых файлов, очистка кэшей.

```bash
make setup-finish
```

### Запуск контейнеров

```bash
make up      # production (запускается по умолчанию)
make up-dev  # development (xdebug / mailpit / logs)
```

## Установка без Docker

**Требования:**
- PHP 8.4+ с расширениями: `mysqli`, `pdo_mysql`, `gd`, `zip`, `intl`, `xsl`, `opcache`, `bcmath`, `pcntl`, `sockets`, `gmp`, `redis`
- Composer 2.x
- MariaDB 10.11+ или MySQL 8+
- Redis 6+
- Nginx 1.20+ или Apache 2.4+

### Подготовка

```bash
cp .env.example .env
composer install
```

### Настройка окружения

- Создайте базу данных и пользователя в MariaDB/MySQL (логин и пароль должны совпадать с указанными в `.env`).
- Убедитесь, что Redis запущен и доступен на `127.0.0.1:6379`.
- Настройте веб-сервер на корень проекта.

### Установка

```bash
make setup-local
```

### Постустановка (обязательно)
  
```bash
make setup-finish
```

## Архитектурная безопасность и защита структуры

В проекте реализована защита `local/php_interface/events/iblock.php` от случайного или несанкционированного изменения структуры данных. Защита автоматически отключается в контексте выполнения миграций `sprint.migration` или CLI-команд.

### Гибридные списки ограничений

Параметры безопасности конфигурируются в файле `local/php_interface/.settings_extra.php` (ключ `hotcom_iblock_restrictions`) и считываются через менеджер конфигурации ядра D7 (`Bitrix\Main\Config\Configuration`):

- **Глобальный флаг** (`$restrictAll = true`) — активирует режим, при котором защищены **абсолютно все** инфоблоки, за исключением тех, что явно указаны в белом списке.
- **Чёрный список** (`$restrictedIblockMasks` / ключ `restricted_masks`) — регулярные выражения (маски) кодов инфоблоков, в которых полностью заблокировано любое изменение структуры.
- **Белый список** (`$allowedIblockMasks` / ключ `allowed_masks`) — регулярные выражения (маски) кодов инфоблоков, структура которых открыта для редактирования при включенном флаге `$restrictAll`.


### Блокировка структуры через события ядра

Контроль изменений выполняется на уровне перехвата событий модулей `main` и `iblock`:

- **Управление разделами:** Запрещено добавление и удаление разделов, а также изменение критических полей `NAME` и `CODE` у существующих разделов.
- **Управление свойствами:** Запрещено создание, изменение параметров и удаление свойств инфоблоков.
- **Управление инфоблоками:** Запрещено создание, изменение конфигурации и удаление самих инфоблоков.
- **Низкоуровневая блокировка типов и UF-полей:** На уровне системного хука `OnPageStart` анализируется контекст выполнения. Полностью блокируются `POST`-запросы на создание, изменение и удаление любых типов инфоблоков (`CIBlockType`) и пользовательских полей (`CUserTypeEntity`).


### Блокировка кастомизации интерфейса

- **Фиксация карточек сущностей:** На уровне события `OnBeforeUserOptionsSet` перехватываются изменения объекта `CUserOptions`. Возможность пользователей и администраторов перестраивать внешний вид детальных форм карточек (вкладки, порядок полей) заблокирована для сохранения унификации интерфейса.
- **Скрытие элементов управления (CSS-инъекции):** При инициализации административных страниц через `Asset::getInstance()->addString()` внедряются глобальные стили, которые физически скрывают из DOM-дерева кнопки создания разделов, добавления свойств, пользовательских полей, а также системную иконку-шестерёнку конфигуратора форм.


## Работа с проектом

В проекте используется основное пространство имён (по умолчанию установлено `Hotcom\\`), которое соответствует директории `local/app/`.

Настроена автоматическая генерация ORM-аннотаций в режиме разработки (`runBitrixOrmAnnotations` в `Hotcom\Helpers\Migration`), генерацию возможно также запустить с помощью команды `make generate-types` в Docker-окружении.

### Структура приложения

```
local/
├── app/                      # Исходный код приложения (PSR-4)
│   ├── Constants/            # Константы
│   ├── Controllers/          # Контроллеры API, наследуют ApiController
│   ├── Dto/                  # Объекты передачи данных
│   ├── Helpers/              # Вспомогательные классы
│   └── Services/             # Бизнес-логика, наследуют AbstractService
├── migrations/               # Миграции (sprint.migration)
├── php_interface/            # Хуки и события Bitrix
│   ├── events/               # Файлы событий
│   └── init.php              # Инициализация
│   └── .settings_extra.php   # Кастомная конфигурация проекта (кэш, ограничения ИБ)
└── routes/                   # Маршрутизация API
```

### Рекомендуемый порядок ознакомления

1. `local/routes/api.php` - маршруты API
2. `local/app/Controllers/` - контроллеры
3. `local/app/Services/` - бизнес-логика
4. `local/app/Helpers` - классы-помощники: кэширование, локализация, нарезка изображений, миграции
5. `local/php_interface/events/` - обработчики событий
6. `local/migrations/`  - миграции

### Маршрутизация

Маршруты API определяются в `local/routes/api.php` с использованием роутинга Bitrix.

### Контроллеры

Контроллеры находятся в `local/app/Controllers/` и наследуют базовый `ApiController`. Все контроллеры обеспечивают:

- Проверку авторизации по токену `Api-Token`;
- Формирование JSON-ответов;
- Обработку ошибок с кодами состояния HTTP.

### Сервисы

Сервисы находятся в `local/app/Services/` и наследуют `AbstractService`. Они обеспечивают:

- Работу с информационными блоками через D7 ORM;
- Кэширование через `Hotcom\Helpers\ApiCache`;
- Поддержку локализации через хелпер `Hotcom\Helpers\Localization`.

### События и хуки

События и хуки находятся в `local/php_interface/events/` и обеспечивают:

- Блокировку / контроль изменений информационных блоков;
- Обработку специфичных случаев;
- Интеграцию с системой кэширования.

### Миграции

Все изменения, касающиеся базы данных, реализуются через библиотеку `sprint.migration`. Миграции хранятся в `local/migrations/`.

- Миграции наследуют класс-помощник `Hotcom\Helpers\Migration`.
- Миграция с базовым функционалом создаётся через административную панель **HC DTO Миграция** (рекомендуется) или с помощью команды `make migration-create`.

### Кэширование
- Глобальное управление поведением кэш-слоя вынесено в изолированный файл конфигурации `local/php_interface/.settings_extra.php`.
- Используется `Hotcom\Helpers\ApiCache` с поддержкой тегированного кэша и Redis.
- Автоматический сброс кэша по ID инфоблока через события `events/cache.php`.

### DTO (Data Transfer Objects)

DTO-классы находятся в `local/app/Dto/` и используются для:

- Передачи данных между слоями приложения;
- Валидации входных данных;
- Структурирования ответов API.

### Безопасность

- Доступ к API защищён токеном `Api-Token`, который задаётся в переменной `BITRIX_API_TOKEN`.
- Защита от нежелательных операций в административной панели.
